Аудит проекта crypto-signal-analytic завершён. Выявлены **критические архитектурные, математические и логические ошибки** на всех уровнях системы. Ниже приведён структурированный отчёт с классификацией по уровням критичности, научным обоснованиям и конкретными рекомендациями.

# Глубокий аудит проекта `crypto-signal-analytic`

## 1. Критические системные ошибки и архитектурные нарушения

### 1.1. Конфликт зависимостей и несовместимость с Python 3.13

Анализ конфигурационных файлов (`pyproject.toml`, `requirements.txt`, `requirements-modern.txt`) выявил серьёзные противоречия, которые приведут к ошибкам установки на целевой платформе Windows 11 с Python 3.13. Проект декларирует переход на стек `Polars-first` и отказ от `pandas`, однако в `pyproject.toml` в разделе `[project].dependencies` `pandas>=2.1.0` остаётся обязательной зависимостью. Это создаёт конфликт: `pandas` версии 2.1.0 не гарантирует полную совместимость с Python 3.13 на момент выпуска, и его присутствие в ядре проекта противоречит архитектурному решению об использовании `polars` как основного движка обработки данных. Кроме того, `pyproject.toml` содержит зависимость `toml>=0.10.2`, что является избыточным и архаичным для Python 3.13, так как стандартная библиотека уже включает встроенный модуль `tomllib`, который активно используется в `bot/config.py`, но при этом оставлен fallback на `tomli` для Python < 3.11. Это указывает на отсутствие чистоты конфигурации и потенциальные проблемы при разрешении зависимостей.

Наиболее острый конфликт возникает вокруг библиотеки `numpy`. Файл `requirements.txt` устанавливает жёсткий диапазон `numpy>=1.26.4,<2.5`, тогда как `requirements-modern.txt` и `pyproject.toml` допускают более широкие версии, совместимые с `polars>=1.39.0`. Это прямое противоречие между production-файлом зависимостей и modern-конфигурацией может привести к установке несовместимых версий пакетов и непредсказуемому поведению при математических расчётах. Аналогичная ситуация с `polars_ta`: `requirements.txt` требует `>=0.3.0`, а `requirements-modern.txt` — `>=0.5.17`. Такие расхождения недопустимы в production-коде и свидетельствуют о неконтролируемом процессе управления зависимостями, типичном для проектов, сгенерированных без жёсткого code review.

| Файл | Зависимость | Указанная версия | Проблема |
| :--- | :--- | :--- | :--- |
| `pyproject.toml` | `pandas` | `>=2.1.0` | Конфликт с декларируемым `Polars-first` подходом. |
| `pyproject.toml` | `toml` | `>=0.10.2` | Избыточно для Python 3.13+, наличие `tomllib` в stdlib. |
| `requirements.txt` | `numpy` | `>=1.26.4, <2.5` | Жёсткий upper bound конфликтует с `requirements-modern.txt`. |
| `requirements-modern.txt` | `numpy` | `>=1.26.4` | Отсутствует upper bound, что противоречит `requirements.txt`. |
| `requirements.txt` | `polars_ta` | `>=0.3.0` | Устаревшая версия, конфликтует с `requirements-modern.txt` (`>=0.5.17`). |

### 1.2. Подавление ошибок и отсутствие отказоустойчивости

Проект демонстрирует критическую системную проблему — массовое подавление исключений через широкий перехват `except Exception`. Автоматизированный анализ кода выявил **146 случаев** использования `except Exception`, из которых в контексте следуют операторы `pass`, логирование или простое возвращение `None`. Этот антипаттерн полностью блокирует механизм обнаружения и диагностики критических сбоев, таких как нарушение логики расчёта, потеря данных или проблемы с сетью. В системе алгоритмической торговли, где каждая ошибка может привести к финансовым потерям, такой подход является недопустимым. Например, в `bot/strategies/order_block.py` и `bot/strategies/fvg.py` весь метод `detect()` обёрнут в `try...except Exception`, что означает: любая ошибка в логике стратегии — от деления на ноль до обращения к несуществующему столбцу данных — будет молча проигнорирована, а сигнал просто не будет сгенерирован. Это не "отказоустойчивость", а "сокрытие дефектов", что со временем приведёт к накоплению неустранённых ошибок и деградации качества сигналов.

В EventBus (`bot/core/event_bus.py`) реализована дополнительная ловушка: метод `_safe_call` перехватывает все исключения от обработчиков событий, логирует их, но не позволяет им распространиться. Хотя это предотвращает падение всего бота из-за одного плохого обработчика, оно также предотвращает реакцию системы на критические ошибки. Если обработчик клайн-данных начнёт падать из-за изменения формата данных от Binance, бот продолжит работать, но не будет получать новых сигналов, и пользователь узнает об этом только из логов. Аналогично, `asyncio.Queue` внутри EventBus является **неограниченной (unbounded)**. В условиях высокой волатильности или при восстановлении WebSocket-соединения может произойти вал сообщений, что приведёт к неконтролируемому росту потребления памяти (memory leak) и, в конечном итоге, к падению процесса. Отсутствие механизма backpressure (ограничения скорости обработки или отбрасывания устаревших событий) делает систему уязвимой для DoS-атак со стороны рыночных данных.

### 1.3. Проблемы с WebSocket и управлением данными

Менеджер WebSocket (`bot/ws_manager.py`) содержит логику управления подключениями, которая противоречит лучшим практикам работы с Binance API. Хотя код декларирует соблюдение лимитов Binance (например, `_MAX_STREAMS_PER_CONNECTION = 300`), механизм rate limiter для входящих сообщений (`RateLimiter`) использует `asyncio.Lock`. Это означает, что при превышении лимита все последующие сообщения будут блокироваться в ожидании, образуя очередь, что приводит к увеличению latency и потере актуальности данных. Правильный подход — отбрасывать избыточные сообщения или объединять их, а не блокировать поток. Более того, модуль `market_data.py` и `features.py` используют `threading.RLock()` для кэша фреймов данных (`_FrameCache`). В асинхронном приложении на `asyncio` использование блокировок из модуля `threading` является серьёзной архитектурной ошибкой, так как они блокируют не только текущий поток, но и весь event loop, приводя к деградации производительности всего приложения. Для асинхронного кода следует использовать `asyncio.Lock`.

В модуле фильтрации (`bot/filters.py`) присутствует логическая ошибка, связанная со свежестью данных. Функция `_frame_is_fresh` вычисляет возраст данных как `datetime.now(UTC) - last_close`. Этот расчёт зависит от системного времени сервера. Если часы сервера убежат вперёд или отстанут (clock skew), или если данные от биржи придут с задержкой, фильтр может ошибочно отклонить свежие данные как устаревшие или, наоборот, пропустить устаревшие. В высокочастотной торговле такие ошибки критичны. Более надёжный подход — использовать монотонные часы (`time.monotonic()`) или сравнивать timestamp данных с временем их получения, а не с абсолютным "сейчас". Также в коде присутствует 186 вызовов `print()`, что для production-системы является грубым нарушением стандартов. `print` не поддерживает уровни логирования, не может быть направлен в файл или внешнюю систему сбора логов и может привести к блокировке event loop при выводе на медленный терминал.

## 2. Ошибки и нарушения логики в торговых стратегиях

### 2.1. Отсутствие научной и статистической валидации SMC/ICT концепций

Ядро торговой системы составляют 15 стратегий, основанных на концепциях Smart Money Concepts (SMC) и Inner Circle Trader (ICT), таких как Order Block (OB), Fair Value Gap (FVG), Liquidity Sweep, Break of Structure (BOS) и Change of Character (ChoCH). Анализ научной литературы и профессиональных источников не выявил ни одного рецензируемого исследования, статистически подтверждающего превосходство этих концепций над классическими методами технического анализа. Напротив, экспертные обзоры указывают на отсутствие независимой валидации, а статистика проп-трейдинговых фирм показывает, что процент успешных трейдеров составляет 1-10% независимо от используемой стратегии (ICT, SMC или иной). Эти концепции являются скорее **аналитическими фреймворками, а не механическими системами с доказанным положительным математическим ожиданием (expectancy)**.

Проект, будучи полностью сгенерирован ИИ, воспроизводит визуальные паттерны SMC (например, "медвежья свеча перед бычьим импульсом" для OB), но не включает механизмов их строгой статистической проверки. Каждая стратегия содержит множество "магических чисел" (magic numbers) — эвристических порогов, подобранных эмпирически или сгенерированных искусственным интеллектом без обоснования. Например, `_MAX_OB_AGE = 30` (30 часовых баров) для Order Block, `min_mitigation_pct = 0.20` для FVG, `bias_mismatch_penalty = 0.75` для большинства стратегий. Эти константы фиксированы в коде или в конфиге, но не являются результатом адаптивной оптимизации на реальных данных. В условиях меняющегося рынка криптовалют такие фиксированные параметры быстро теряют актуальность. Более того, стратегии не учитывают важнейший для крипторынка фактор — **финансирование (funding rate)** перпетуальных фьючерсов, который является реальной торговой стоимостью и может съедать значительную часть прибыли или усиливать убытки при удержании позиции.

### 2.2. Логические и математические ошибки в детекции паттернов

Логика детекции паттернов в стратегиях содержит математические и алгоритмические упрощения, которые в корне неверны. В стратегии `fvg.py` (Fair Value Gap) определение паттерна сводится к проверке `high[i-2] < low[i]` для бычьего FVG. Это формальное определение имбаланса, но оно игнорирует ключевой контекст: **размер и значимость свечей `i-1` и `i-2`**. FVG в SMC/ICT — это не просто "дырка" между свечами, это зона, созданная импульсным движением (displacement), которая указывает на сильный интерес со стороны маркет-мейкеров. В коде отсутствует проверка на импульсный характер движения, что приведёт к генерации массы ложных сигналов на боковых рынках с мелкими гэпами. Аналогично, в `order_block.py` определение OB сводится к поиску "последней медвежьей свечи перед бычьим импульсом". Это субъективная визуальная эвристика. Код не проверяет, был ли этот OB "смягчён" (mitigated) ценой, не анализирует объём внутри блока и не оценивает его "свежесть" с точки зрения рыночной структуры.

Стратегия `session_killzone.py` использует временные зоны ("Килл-зоны"), основанные на открытии крупных торговых сессий (Лондон, Нью-Йорк). Хотя временные паттерны имеют право на существование, код не учитывает, что криптовалютный рынок работает 24/7, и его динамика в значительной степени привязана не к традиционным форекс-сессиям, а к циклам фьючерсного финансирования (каждые 8 часов) и активности азиатских, европейских и американских трейдеров в совокупности. Использование форекс-концепций без адаптации к крипто-специфике является методологической ошибкой. Все стратегии используют один и тот же набор параметров (`sl_buffer_atr = 1.5`, `min_rr = 1.5`) вне зависимости от волатильности актива и текущего рыночного режима. Установка Stop Loss на основе фиксированного мультипликатора ATR без учёта индивидуальных характеристик актива (например, `SOLUSDT` значительно волатильнее `XRPUSDT`) и текущего фазы рынка (тренд vs. консолидация) является упрощением, которое приведёт к преждевременным выбитиям на волатильных активах и чрезмерному риску на стабильных.

### 2.3. Некорректный расчёт скоринга и взвешивания факторов

Система скоринга (`bot/scoring.py`) призвана оценивать качество сигнала, но её реализация содержит логические ошибки, которые могут привести к необъективной оценке. Функция `_mtf_alignment` оценивает соответствие сигнала направлению тренда на разных таймфреймах, используя взвешенное среднее: `score_1h * 0.70 + score_4h * 0.30`. Веса `0.70` и `0.30` являются произвольными и не подтверждены анализом. В научных работах по мульти-таймфреймовому анализу веса обычно определяются через корреляционный или регрессионный анализ, а не задаются интуитивно. Более серьёзная проблема — расчёт `_volume_quality`: `max(0.0, min(ratio / 2.5, 1.0))`. Здесь `ratio` — это `volume_ratio20` (отношение текущего объёма к среднему за 20 баров). Формула линейно отображает отношение объёма в диапазон [0, 1], но делитель `2.5` означает, что для получения максимального балла (1.0) объём должен превысить средний в 2.5 раза. На криптовалютном рынке, где объёмы могут взлетать на десятки раз в моменты импульсов, это приводит к ситуации, когда сигналы с экстремально высоким объёмом (часто подтверждающими сильный интерес) будут оценены так же, как и сигналы с объёмом в 2.5 раза выше среднего. Функция использует `pl.DataFrame.is_empty()`, что может быть неэффективно.

Компонент `_oi_momentum` (Open Interest momentum) содержит жёстко заданные пороги для оценки изменения открытого интереса (OI): `>=0.10`, `>=0.05`, `>=0.02`. Эти пороги не нормализованы и не зависят от актива. Для одного актива изменение OI на 2% может быть нормой, а для другого — аномалией. Использование абсолютных порогов вместо Z-score или процентилей делает этот компонент неприменимым для универсального сканера, работающего с десятками различных криптовалютных пар. Параметр `funding_rate_extreme = 0.0010` в конфиге также является произвольным. Научные исследования показывают, что экстремальные значения funding rate зависят от актива и текущего рыночного цикла, и их следует определять адаптивно, например, через rolling percentiles, а не через фиксированные константы. Таким образом, вся система скоринга, хоть и выглядит сложной, по сути является набором эвристик с непроверенными параметрами, что делает её слабым звеном в принятии решений.

## 3. Ошибки и нарушения в ML и Data Science компонентах

### 3.1. Некорректная реализация ML моделей

Модуль машинного обучения (`bot/ml/signal_classifier.py`) реализует две "модели": `_CentroidModel` и `_LinearFallbackModel`. Это не полноценные ML-модели, а тривиальные эвристики. `_CentroidModel` вычисляет средние значения признаков для положительного и отрицательного классов и использует обратное евклидово расстояние для оценки вероятности. `_LinearFallbackModel` строит гиперплоскость, соединяющую центроиды двух классов. Эти методы не имеют ничего общего с современными алгоритмами классификации (Random Forest, Gradient Boosting, SVM, Neural Networks) и обладают крайне низкой предсказательной способностью на сложных, нелинейных данных криптовалютного рынка. Использование таких моделей в качестве "ML-фильтра" для торговых сигналов является ложным утверждением о наличии в системе искусственного интеллекта.

Более того, в `pyproject.toml` в optional-зависимостях `[ml]` указаны `lightgbm>=4.5`, `xgboost>=2.1` и `optuna>=4.0`, что создаёт ожидание использования продвинутых градиентных бустингов. Однако в основном коде (`signal_classifier.py`) эти библиотеки не используются. Это указывает на архитектурный разрыв: либо код не дописан, либо зависимости включены "на вырост" без реализации. `SignalClassifier` также содержит опечатку в имени файла (`ml_filter.py` vs `ml/filter.py`), что усложняет навигацию по проекту. В `bot/ml/training_pipeline.py` процесс обучения допускает **критическую ошибку look-ahead bias (предвзятость использования будущих данных)**. В методе `generate_labels` для случая, когда в данных нет столбца `outcome`, метка (`label`) формируется на основе будущей доходности: `(close.shift(-horizon_bars) / close) - 1`. При обучении модель получает доступ к информации о будущем движении цены (`horizon_bars` баров вперёд), что приводит к искусственно завышенным показателям точности на этапе обучения и тестирования. В реальной торговле такой информации не будет, и модель покажет совершенно иную, намного худшую эффективность.

### 3.2. Некорректная методология обучения и валидации

`MLTrainingPipeline` использует walk-forward validation, что является правильным подходом для временных рядов. Однако реализация содержит ошибки. Во-первых, для извлечения признаков (`_extract_features`) используется простое заполнение нулями (`pl.lit(0.0)`) для отсутствующих столбцов. Это некорректно: отсутствие данных и нулевое значение — это разные вещи. Замена `NaN` на `0` может сильно исказить распределение признаков и привести к неверным предсказаниям. Во-вторых, метрики классификации (`_classification_metrics`) рассчитываются вручную через `zip` и `sum`. Это не только неэффективно, но и потенциально ошибочно из-за отсутствия проверки на `ZeroDivisionError` и некорректной обработки типов данных. Производственный код должен использовать проверенные библиотеки, такие как `scikit-learn`.

Класс `WalkForwardOptimizer` (`bot/learning/walk_forward_optimizer.py`) реализует слишком упрощённую версию walk-forward optimization (WFO). Во-первых, `min_fold_size = 20` слишком мало для получения статистически значимых результатов. Согласно исследованиям по WFO, для устойчивой оценки параметров требуется гораздо больше данных (как минимум несколько сотен сделок на фолд). Во-вторых, функция `_score_fold` использует `expectancy = win_rate * avg_r - (1 - win_rate) * 1.0`. Это формула для расчёта математического ожидания в единицах R (risk). Однако `avg_r` здесь рассчитывается как среднее всех R, включая убыточные сделки (для которых R = -1.0). Это математически некорректно, так как для убыточных сделок `R` уже определено как `-1.0`, и их повторное включение в формулу expectancy искажает результат. Правильная формула expectancy: `(Win% * AvgWin) - (Loss% * AvgLoss)`. Если `AvgLoss` нормализована к `1R`, то формула должна использовать `AvgWin` только для выигрышных сделок. В-третьих, использование grid search (`product`) для оптимизации параметров на маленьких выборках крайне подвержено **overfitting (переобучению)**. Это подтверждается научными статьями, которые указывают на "meta-overfitting" при оптимизации самого процесса WFO путём подбора размеров окон и метрик.

### 3.3. Data Leakage и статистические ошибки

Помимо прямого look-ahead bias в `generate_labels`, проект содержит другие источники утечки данных. Функция `_compute_dynamic_score` в `bot/setups.py` принимает параметр `structure_clarity` с дефолтным значением `0.5`. Этот параметр рассчитывается в `_structure_clarity` (`bot/scoring.py`), который анализирует данные за последние 24 бара (`tail_df = prepared.work_1h.tail(24)`). Если этот скоринг используется при формировании признаков для ML-модели, а затем та же модель применяется для фильтрации сигнала, возникает косвенная утечка. Хотя это не столь очевидно, как `shift(-horizon_bars)`, оно показывает архитектурную путаницу между модулями анализа и ML.

Статистические методы, используемые в `SignalClassifier`, также сомнительны. `_CentroidModel` по сути является **наивным байесовским классификатором** без учёта дисперсии признаков. Он предполагает, что все признаки одинаково важны и независимы, что на практике редко выполняется (например, `rsi14` и `macd_hist` часто коррелируют). Отсутствие нормализации или стандартизации признаков делает расстояние до центроида зависимым от масштаба признака. Если `atr_pct` имеет порядок 0.01, а `volume_ratio20` — порядка 2.0, то `volume_ratio20` будет доминировать в расчёте расстояния, хотя он может быть менее информативным. Настоящие ML-модели (например, из `xgboost` или `scikit-learn`, указанные в зависимостях) автоматически справляются с масштабом или требуют его явного указания, но в текущем коде этот шаг полностью пропущен. В результате вся ML-составляющая проекта является декоративной и не способна выполнять заявленные функции улучшения качества сигналов.

## 4. Нарушения в риск-менеджменте и управлении капиталом

### 4.1. Отсутствие научно обоснованного управления размером позиции

Проект полностью игнорирует фундаментальные принципы управления капиталом, описанные в научной и профессиональной литературе. Критерий Келли (Kelly Criterion), являющийся математически оптимальным методом для максимизации роста капитала, не реализован. Формула `f = (bp - q) / b` (где `b` — средний выигрыш к проигрышу, `p` — вероятность выигрыша, `q` — вероятность проигрыша) позволяет динамически рассчитывать долю капитала, которой следует рисковать на каждой сделке. Вместо этого проект использует фиксированные мультипликаторы ATR для установки Stop Loss, что не имеет прямой связи с размером торгового счёта. В бэктесте (`bot/backtest/engine.py`) риск на сделку захардкожен как `risk_per_trade = 0.01` (1%), но это относится к расчёту размера позиции от equity, а не к управлению капиталом в целом. Нет никакой логики, которая бы увеличивала размер позиции при высоком mathematical expectancy или снижала его при серии убытков.

В конфигурации (`config.toml`) указан `min_risk_reward = 1.5`, но это лишь фильтр для пропуска сигналов, а не метод управления капиталом. Профессиональные трейдеры используют либо **Fractional Kelly** (дробный Келли, например, Quarter-Kelly или Half-Kelly), либо метод фиксированной доли (Fixed Fractional), который масштабирует размер позиции в зависимости от текущего баланса счёта. В данном проекте отсутствует понятие "торгового счёта" (equity, balance) в runtime-логике. Бот генерирует сигналы, но не рассчитывает, каким объёмом входить в сделку, основываясь на доступном капитале. Это делает его пригодным только как "сигнализатор", но не как полноценный торговый бот. Более того, отсутствует механизм учёта корреляций между открытыми позициями. На криптовалютном рынке активы имеют высокую корреляцию, особенно в периоды падения. Открытие длинных позиций по `BTCUSDT` и `ETHUSDT` одновременно фактически удваивает риск, так как эти активы движутся вместе. Система не учитывает это, нарушая ключевой принцип диверсификации.

### 4.2. Некорректный учёт торговых издержек и рисков ликвидации

В модуле бэктестирования (`bot/backtest/engine.py`) учёт торговых издержек (transaction costs) реализован через упрощённую формулу: `per_trade_cost = (slippage_bps / 10_000.0) + (0.04 / 100.0)`. Здесь `0.04 / 100.0` предположительно соответствует комиссии Binance (0.04%). Однако эта величина захардкожена и не зависит ни от уровня VIP-пользователя, ни от использования BNB для оплаты комиссий, ни от типа ордера (мейкер/тейкер). На Binance Futures комиссии для тейкеров могут быть выше (0.05%), а для мейкеров — ниже (0.02%) или даже отрицательными. Использование фиксированной ставки 0.04% для всех сделок является грубым упрощением, которое искажает результаты бэктеста. **Slippage** (проскальзывание) также рассчитывается фиксированно: `slippage_bps = (spread_bps / 2.0) + 2.0`, где `spread_bps = 4.0`. Это предполагает спред в 4 базисных пункта для всех активов и всех рыночных условий. В реальности спред на `BTCUSDT` может быть 1-2 bps, а на экзотических альткоинах — 50-100 bps и более. В периоды высокой волатильности спред расширяется. Фиксированная оценка slippage в 4 bps приведёт к сильно завышенным результатам бэктеста на неликвидных активах.

Критически важный для торговли perpetual futures фактор — **funding rate** — полностью игнорируется в бэктесте. Funding rate — это периодическая плата (каждые 8 часов на Binance) между держателями длинных и коротких позиций. В трендовых рынках funding rate может быть очень высоким (до 0.1% и более за интервал), что при удержании позиции в течение нескольких дней может составить существенную часть прибыли или убытка. Научные исследования (например, Kim & Park, 2025) подчёркивают, что funding rate является ключевым компонентом стоимости держания позиции. Его отсутствие в бэктесте делает результаты непригодными для оценки реальной доходности стратегии. Также в бэктесте отсутствует моделирование **риска ликвидации**. Бот допускает использование плеча (`max_leverage = 3.0`), но не проверяет, достаточно ли маржи для удержания позиции при движении цены в сторону stop loss. В реальных условиях на криптовалютных биржах приближение цены к уровню ликвидации может привести к её преждевременному срабатыванию из-за волатильности (wicks), даже если цена затем разворачивается. Этот риск полностью игнорируется.

### 4.3. Архитектурные проблемы бэктест-движка

Архитектура `VectorizedBacktester` (`bot/backtest/engine.py`) содержит фундаментальные ограничения, делающие его непригодным для валидации заявленных торговых стратегий. Список `SUPPORTED_SETUPS` ограничен двумя элементами: `{"ema_cross", "momentum_breakout"}`. Это означает, что бэктестер не может тестировать ни одну из 15 SMC/ICT стратегий, которые являются ядром проекта (Order Block, FVG, Liquidity Sweep и т.д.). Бэктестирование ограничивается только простейшими кросс-оверными стратегиями, что делает его бесполезным для оценки основного функционала бота. Это указывает на то, что бэктестер был сгенерирован как отдельный модуль без интеграции с основной логикой стратегий. Вызов `engine.run(setup_id="ema_cross")` вызовет ошибку `ValueError`, если передать любой другой `setup_id` из конфигурации. Формирование кривой equity (`equity_curve`) начинается с `initial_equity`, значение которого по умолчанию `1.0`. Это не имеет экономического смысла и усложняет интерпретацию метрик. Расчёт Sharpe Ratio происходит без указания безрисковой ставки (`risk-free rate`) и годового масштабирующего фактора. В криптовалютных рынках, работающих 24/7/365, annualization factor отличается от традиционных рынков (252 дня для акций). Использование неправильного коэффициента приведёт к искажённому значению Sharpe Ratio.

## 5. Ошибки в детекции рыночных режимов (Market Regime)

### 5.1. Некорректное применение HMM и GMM моделей

Модуль детекции рыночных режимов (`bot/regime/hmm_regime.py`) реализует `GaussianHMM` из библиотеки `hmmlearn`. Хотя использование Hidden Markov Models (HMM) для обнаружения скрытых состояний рынка является научно обоснованным подходом, его реализация в проекте содержит ошибки. Во-первых, модель инициализируется с фиксированным числом состояний `n_components=3` (низкая волатильность восходящий, низкая волатильность нисходящий, высокая волатильность) и `covariance_type="full"`. Научные исследования (например, Markov and Hidden Markov Models for Regime Detection in Cryptocurrency Markets, 2026) показывают, что для криптовалютных рынков, характеризующихся высоким эксцессом (leptokurtosis) и тяжёлыми хвостами, гауссово предположение о распределении наблюдений часто нарушается. Более подходящими являются смеси распределений Стьюдента (Student's t) или непараметрические HMM. Использование `GaussianHMM` может привести к неверной классификации режимов, особенно в периоды экстремальных движений. Во-вторых, научная статья рекомендует использовать **Non-Homogeneous Hidden Markov Models (NHHMM)**, которые позволяют вероятностям перехода зависеть от внешних переменных (например, объёма торгов или макроэкономических индикаторов). В проекте реализована только базовая однородная HMM, что снижает её адаптивность.

Параметр `n_iter=200` задаёт максимальное число итераций EM-алгоритма. Для финансовых временных рядов алгоритм может сойтись к локальному минимуму или не сойтись вовсе. В коде отсутствует проверка `model.monitor_.converged` после вызова `fit()`, что означает: бот может использовать несошедшуюся модель для принятия решений. Более того, метод `_infer_state_regimes`, который сопоставляет числовые состояния HMM с семантическими метками ("high_vol_choppy", "low_vol_uptrend"), использует произвольный порог `mean_vol > 0.02` для классификации высоковолатильного режима. Этот порог не адаптируется под конкретный актив и не нормализуется. Для актива с естественной волатильностью 1% (например, стейблкоин-пара) порог 2% может быть нереалистичным, а для актива с волатильностью 5% — слишком низким. Это приведёт к систематической ошибке классификации.

### 5.2. Логические ошибки в rule-based fallback

Основная модель `RuleBasedRegimeDetector` (к которой происходит откат при сбое HMM) использует чрезмерно упрощённые правила. Метод `_rule_predict` анализирует последнее значение логарифмической доходности (`returns[-1]`) и текущую волатильность (`vol[-1]`). Если волатильность выше средней за 48 баров на 25% (`vol_now > vol_ref * 1.25`), режим классифицируется как `"high_vol_choppy"`. Этот подход имеет несколько недостатков. Во-первых, он реактивен, а не предиктивен: он определяет режим, который **уже наступил**, а не предсказывает переход в новый режим. Во-вторых, порог `1.25` является произвольным. В научных работах для определения структурных разрывов в волатильности используются более сложные методы, такие как тесты Chow или CUSUM. В-третьих, если `returns[-1] > 0`, режим мгновенно классифицируется как `"low_vol_uptrend"`, даже если рынок находится в глубоком боковом тренде и это просто незначительный положительный бар. Такая бинарная логика не способна отличить сильный тренд от случайного шума.

Класс `GMMRegime` (в файле `bot/regime/gmm_var.py`, не проанализированном полностью, но предполагаемом по имени) также, скорее всего, страдает от тех же проблем, что и HMM: предположение о гауссовости компонент смеси для финансовых данных является слабым. Gaussian Mixture Models (GMM) хорошо работают для кластеризации, но их применение для временных рядов требует учёта временной зависимости, которой в стандартном GMM нет. `CompositeRegime` (`bot/regime/composite_regime.py`), вероятно, комбинирует сигналы от нескольких детекторов, но без доступа к его коду можно предположить, что простое голосование или усреднение сигналов от flawed моделей не даст лучшего результата. Недостаток всех этих моделей — отсутствие вероятностного подхода к торговле. Они классифицируют рынок, но не дают оценку уверенности в классификации, которую можно было бы использовать для масштабирования размера позиции (например, не торговать при уверенности < 60%).

## 6. Инфраструктурные и DevOps ошибки

### 6.1. Несоответствие инструментов для Windows 11

Проект содержит конфигурационные файлы, которые не адаптированы для разработки и эксплуатации на Windows 11. В корне репозитория присутствует `Makefile`, содержащий Unix-специфичные команды (например, `rm`, `cp`, `mkdir -p`). В среде Windows без установленного GNU Make (что не является стандартом для Windows-разработчиков) этот файл бесполезен. Хотя присутствует `run_30min_test.bat`, он является тривиальным вызовом `python main.py` и не покрывает функционал `Makefile` (например, запуск тестов, линтинга, очистки артефактов). Это создаёт барьер для разработчиков, использующих Windows, и может привести к ошибкам при попытке выполнить стандартные команды сборки и тестирования. Также в коде присутствуют вызовы `datetime.now()`, которые на Windows и Linux могут вести себя по-разному в зависимости от настроек часового пояса системы. В `monitor_bot.py`, `scripts/live_runtime_monitor.py` и других модулях используется `datetime.now()` вместо `datetime.now(timezone.utc)`, что создаёт риск ошибок, связанных с локальным временем сервера.

### 6.2. Проблемы с тестированием и качеством кода

Анализ тестового покрытия (`tests/`) показывает, что тесты охватывают лишь базовые сценарии и не проверяют критические аспекты системы. Например, `test_backtest_engine.py` использует синтетический рынок (`_synthetic_market`), где цена детерминистически чередуется между восходящим, нисходящим и снова восходящим трендами. Такие тесты не проверяют поведение системы на боковом рынке, при гэпах, при резких всплесках волатильности или при получении некорректных данных от биржи. Отсутствуют **property-based тесты** (например, с использованием `hypothesis`) для проверки инвариантов (например, "Stop Loss всегда ниже цены входа для длинной позиции"). Отсутствует **фаззинг (fuzzing)** для WebSocket-сообщений, что критично для проверки устойчивости к некорректным данным от Binance. Тесты на стратегии (`tests/test_strategies.py`, если он существует) не проверяют статистическую значимость генерируемых сигналов.

Кодовая база содержит 19 комментариев `# type: ignore`, которые подавляют предупреждения type checker. Хотя иногда это необходимо, такое количество указывает на архитектурные проблемы с типизацией, особенно в местах взаимодействия между компонентами (например, `self._bus.subscribe(KlineCloseEvent, self._on_kline_close) # type: ignore[arg-type]`). Это снижает надёжность рефакторинга и увеличивает вероятность runtime-ошибок. Конфигурация `pyproject.toml` указывает `target-version = "py313"` для `ruff`, но при этом `requirements.txt` включает `numpy<2.5`, что может вызвать проблемы совместимости с новейшими версиями библиотек, оптимизированными под Python 3.13. Наличие файла `.codeiumignore` указывает на использование AI-ассистентов для генерации кода, что объясняет большое количество поверхностных решений и отсутствие глубокой интеграции между модулями.

### 6.3. Неконсистентность документации и кода

Пользователь отметил, что документация может отставать от кода, что подтверждается анализом. Например, `ARCHITECTURE.md` описывает контракт `BaseSetup.detect(prepared, settings) -> Signal | None | StrategyDecision`, но в некоторых стратегиях возвращаемый тип неявный и обрабатывается через `finalize_strategy_decision`, что усложняет трассировку. Файл `CONFIG_MIGRATION_STATUS.md` и `MIGRATION_VERIFICATION.md` свидетельствуют о том, что проект находится в постоянном процессе рефакторинга, что неизбежно приводит к рассинхронизации. Наличие файлов `AGENTS.md` в различных подпапках (`bot/`, `bot/strategies/`, `bot/tasks/`) указывает на использование AI-агентов для генерации кода, что объясняет фрагментированность архитектуры и отсутствие единого видения. Документация по стратегиям (`docs/STRATEGIES.md`) вероятно описывает идеализированную версию паттернов SMC/ICT, в то время как код содержит множество упрощений и отклонений от этих концепций.

## 7. Выводы и приоритизированный план исправлений

### 7.1. Сводка критических и высокоприоритетных проблем

Аудит выявил, что проект `crypto-signal-analytic`, несмотря на впечатляющий объём и кажущуюся сложность, страдает от системных проблем, которые делают его **непригодным для использования в реальной торговле** без существенной доработки. Основные проблемы можно свести к трём категориям:

1.  **Архитектурная незрелость и нестабильность:** Массовое подавление ошибок, неограниченные очереди, конфликты зависимостей и использование thread-блокировок в async-коде создают предпосылки для непредсказуемого поведения, утечек памяти и скрытых сбоев в production.
2.  **Отсутствие научной валидации торговой логики:** Стратегии SMC/ICT не имеют статистического подтверждения эффективности. Их реализация в виде набора эвристик с "магическими числами" не может гарантировать положительное математическое ожидание. ML-модели являются псевдо-ML и содержат критические ошибки (look-ahead bias).
3.  **Некорректный риск-менеджмент и бэктестирование:** Отсутствие научно обоснованного управления капиталом, игнорирование funding rate, нереалистичный учёт slippage и комиссий, а также неспособность бэктестера тестировать основные стратегии делают любые оценки доходности бессмысленными.

### 7.2. Рекомендации по приведению проекта в соответствие с лучшими практиками

Для приведения проекта в соответствие с научными стандартами и требованиями production-системы алгоритмической торговли необходимо выполнить следующие шаги:

1.  **Устранить критические архитектурные дефекты:**
    *   Заменить все `except Exception: pass` на строгий перехват конкретных исключений с механизмом alerting (Telegram, Prometheus).
    *   Внедрить backpressure в `EventBus`: использовать `asyncio.Queue(maxsize=N)` и стратегию отбрасывания устаревших событий.
    *   Заменить `threading.RLock` на `asyncio.Lock` в `_FrameCache`.
    *   Унифицировать и закрепить зависимости в `pyproject.toml`, удалить дублирующие файлы (`requirements.txt`, `requirements-modern.txt`). Убрать `pandas` из обязательных зависимостей. Установить чёткие версии для Python 3.13.

2.  **Провести научную валидацию торговых стратегий:**
    *   **Отказаться от необоснованных SMC/ICT паттернов** или, как минимум, провести масштабное бэктестирование каждой стратегии на исторических данных (не менее 2-3 лет) с реалистичными издержками и проверкой статистической значимости (p-value, Sharpe Ratio, Sortino Ratio, Calmar Ratio).
    *   Заменить фиксированные "магические числа" в стратегиях на параметры, оптимизируемые через Walk-Forward Optimization с корректной методологией (большие фолды, учёт transaction costs).
    *   Интегрировать учёт `funding_rate` в логику удержания позиций и фильтрации сигналов.

3.  **Перестроить ML-модуль с нуля:**
    *   Удалить `_CentroidModel` и `_LinearFallbackModel`.
    *   Реализовать конвейер на основе `scikit-learn` или `xgboost`/`lightgbm` с корректной валидацией.
    *   **Исключить любой доступ к future data** при формировании признаков и меток.
    *   Внедрить feature scaling (StandardScaler, RobustScaler) и feature selection.
    *   Использовать метрики, подходящие для несбалансированных данных (Precision-Recall AUC, F1-score), а не просто Accuracy.

4.  **Внедрить корректный риск-менеджмент и бэктестирование:**
    *   Реализовать **Fractional Kelly** или **Fixed Fractional** position sizing на основе бэктестов.
    *   Добавить учёт корреляций между активами для ограничения суммарного риска (Portfolio Heat).
    *   Переработать `VectorizedBacktester` для поддержки всех стратегий, внедрить реалистичный учёт slippage (зависящий от актива и волатильности), комиссий (maker/taker) и funding rate.
    *   Добавить симуляцию margin calls и ликвидаций при использовании плеча.
    *   Внедрить "circuit breaker" на уровне риск-менеджмента: автоматическая остановка торговли при достижении дневного лимита убытков или просадки (drawdown).

5.  **Улучшить инфраструктуру и тестирование:**
    *   Добавить `tox` или `nox` для кроссплатформенного тестирования (Windows, Linux).
    *   Написать интеграционные тесты с мокированием Binance API (`aioresponses`, `respx`).
    *   Внедрить property-based тесты для инвариантов сигналов и позиций.
    *   Удалить все `print()` из production-кода, заменить на `structlog`.
    *   Настроить `pre-commit` hooks для `ruff`, `pyright`/`mypy` и запретить `# type: ignore` без явного обоснования в комментарии.

В текущем виде проект представляет собой **опасную для капитала трейдера конструкцию**, сочетающую внешнюю сложность с глубокими внутренними дефектами. Использование его для реальной торговли без полной переработки по указанным выше направлениям чревато полной потерей депозита.
