# Единый итоговый документ по проходу v2

Дата: 2026-04-28
Источник: консолидация `docs/REMEDIATION_PLAN.md` и `docs/remediation_ledger.md`.

## 1) Критично

### C1. Live-ML guardrails остаются незавершёнными (baseline fallback safety)
- **Где найдено:** `docs/REMEDIATION_PLAN.md` (Task 3 = In progress), `docs/remediation_ledger.md` (live ML fallback закрыт частично).
- **Риск:** модельная деградация в проде может происходить без ожидаемого уровня качества сигналов при переключении на baseline-классификатор.
- **Влияние:** ложные сигналы/пропуск сигналов, ухудшение PnL и доверия к системе фильтрации в live-режиме.
- **Критерий исправления:**
  1. Любой baseline-классификатор (`centroid_baseline`, `linear_baseline`) в live гарантированно переводит ML-фильтр в explicit-disable.
  2. В телеметрии фиксируется `model_kind`, причина disable и частота срабатывания guardrail.
  3. В regression/integration тестах есть отдельные кейсы для live-disable и для offline-режима.
- **Статус верификации:**
  - ✅ **Подтверждено статически:** контракт и статус «в процессе» отражены в плане/ledger.
  - ⚠️ **Гипотеза до runtime-проверки:** в реальной сессии guardrails всегда срабатывают корректно на всех путях загрузки модели.

**Инженерная задача T1: Довести ML guardrails до production-grade**
1. Проследить все live code paths выбора классификатора и унифицировать точку принятия решения `is_live && is_baseline => disable`.
2. Добавить структурированную телеметрию (model kind, disable reason, stage).
3. Добавить тесты: unit (classifier selection), integration (end-to-end filter behavior), regression (no silent fallback).
4. Провести runtime dry-run (paper/live-sim) и снять артефакты по guardrail-событиям.

  - ✅ **Подтверждено статически:** в ledger явно отмечено как незавершённое follow-up.
  - ⚠️ **Гипотеза до runtime-проверки:** после quarantine не возникнут деградации в долгоживущих сценариях.

**Инженерная задача T2: Завершить feature-contract cleanup и quarantine scaffolds**

## 2) Средне

### M1. Декомпозиция `ws_manager` не завершена
- **Где найдено:** `docs/REMEDIATION_PLAN.md` (`ws_manager decomposition = In progress`; остаются connection/subscriptions extraction).
- **Риск:** локализация дефектов в websocket-ветке дороже; выше шанс побочных эффектов при изменениях.
- **Влияние:** рост времени на поддержку и повышенный риск регрессий в каналах подписок/переподключений.
- **Критерий исправления:**
  1. Разделение на модули `connection`, `subscriptions`, `cache`, `enrichment` доведено до конца.
  2. Явно сохранён и проверен split `rest_full` vs `ws_light` + fallback telemetry.
  3. Регрессии по websocket-потокам проходят в целевых тестах.
- **Статус верификации:**
  - ✅ **Подтверждено статически:** статус и remaining scope явно зафиксированы в плане.
  - ⚠️ **Гипотеза до runtime-проверки:** устойчивость reconnect/subscribe under load без деградации latency.

**Инженерная задача T3: Финализировать декомпозицию websocket-слоя**
1. Выделить connection/subscription-логику в отдельные модули.
2. Перенести зависимости по контрактам, минимизируя циклические импорты.
3. Добавить targeted tests на reconnect, resubscribe, fallback ветки и telemetry.
4. Выполнить runtime smoke на живом потоке market-data.

### M2. Декомпозиция `features` не завершена
- **Где найдено:** `docs/REMEDIATION_PLAN.md` (`features decomposition = In progress`; остаются core/advanced/oscillator splits).
- **Риск:** высокая сложность модуля признаков затрудняет безопасные изменения.
- **Влияние:** замедление внедрения новых сигналов и рост вероятности cross-impact регрессий.
- **Критерий исправления:**
  1. Модуль разделён на тематические подпакеты с явными контрактами.
  2. Нет скрытых сайд-эффектов/неявных зависимостей между блоками признаков.
  3. Покрытие тестами по группам признаков актуализировано.
- **Статус верификации:**
  - ✅ **Подтверждено статически:** остаточный scope указан в плане.
  - ⚠️ **Гипотеза до runtime-проверки:** производительность расчёта признаков не ухудшилась в live pipeline.

**Инженерная задача T4: Завершить модульную декомпозицию feature-engineering**
1. Вынести `core`, `advanced`, `oscillators` в отдельные модули и зафиксировать API.
2. Убрать дубли и централизовать общие helper-функции.
3. Обновить/добавить тесты на паритет результатов «до/после».
4. Прогнать бенчмарк hot-path расчётов на representative data.

## 3) Улучшения

### I1. Декомпозиция тестов и поддерживаемость regression-сета
- **Где найдено:** `docs/REMEDIATION_PLAN.md` (Task 6 = In progress).
- **Риск:** избыточно крупные тестовые файлы усложняют triage падений.
- **Влияние:** увеличение времени диагностики и код-ревью.
- **Критерий исправления:**
  1. Regression-набор разделён по тематическим сьютам.
  2. Каждый сьют покрывает одну функциональную область с минимальной связностью.
  3. CI-отчёт позволяет быстро локализовать проблему по имени сьюта.
- **Статус верификации:**
  - ✅ **Подтверждено статически:** статус «в процессе» зафиксирован.
  - ℹ️ **Гипотеза до runtime-проверки:** сокращение MTTR по инцидентам тестов подтвердится только на нескольких итерациях CI.

**Инженерная задача T5: Оптимизировать структуру regression-тестов**
1. Дорезать крупные сьюты на подмодули по предметным областям.
2. Ввести единый шаблон именования тестовых файлов/классов.
3. Добавить маркеры и таргетные команды запуска для ускоренного triage.

- Есть минимум два критичных незакрытых пункта (C1, C2), влияющих на безопасность live-решений и устойчивость runtime-контрактов.
